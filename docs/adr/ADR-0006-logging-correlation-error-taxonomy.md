# ADR-0006: Logging, Correlation ID, and Error Taxonomy

## Context

Three cross-cutting concerns require a shared specification across all
services:

1. **Logging schema**: every log record across every service shares a
   minimum schema; field names are consistent; PII redaction (XR-005) is
   universal.
2. **Correlation ID propagation**: a partner record traverses 6+
   asynchronous service boundaries (ingestion → mapping → DQ → identity
   resolution → canonical update → audit emit; later, a Verification call
   reads the same canonical state). A stable correlation ID threading
   through every log entry, every database row, every audit event, every
   Pub/Sub message is the only viable debuggability primitive.
3. **Error taxonomy**: every service handles errors. Without a shared
   taxonomy (transient vs. permanent, retryable vs. not, alert vs. quarantine),
   each service invents its own — leading to inconsistent retry behavior,
   inconsistent alerting, inconsistent error reporting to callers.

Without these, the system is technically correct but operationally
unobservable.

## Decision

### 1. Structured logging schema

All services use `structlog` configured per
`src/lore_eligibility/bootstrapper/logging_config.py` (already present from
Phase 00 harness). The PII redaction processors (already implemented) run
before any renderer.

Every log record carries the following required fields:

| Field | Type | Source | Example |
|---|---|---|---|
| `timestamp` | ISO-8601 UTC | structlog `TimeStamper` | `2026-04-29T17:31:42.018Z` |
| `severity` | string | log call | `INFO` / `WARNING` / `ERROR` / `CRITICAL` |
| `event` | string | log call | `"member lookup completed"` |
| `service_name` | string | bound at startup | `"verification-api"` |
| `service_version` | string | bound at startup | git-SHA |
| `environment` | string | bound at startup | `"prod"` / `"staging"` / `"dev"` |
| `trace_id` | string (UUID/W3C) | propagated | (per W3C Trace Context) |
| `span_id` | string | propagated | (per W3C Trace Context) |
| `correlation_id` | string (UUID) | propagated | per-record ID; see §2 |

Optional fields (bound at logger creation when applicable):
- `partner_id` (for partner-scoped operations)
- `member_id` or `target_token` (tokenized; never plaintext PII)
- `actor_role` (for audit-relevant operations)
- `request_id` (for HTTP request scope)

A shared factory in `shared/telemetry.py` provides bound loggers:

```python
from lore_eligibility.shared.telemetry import get_logger

logger = get_logger(__name__)  # returns a structlog logger pre-bound with
                                # service_name, service_version, environment
```

Direct `logging.getLogger()` use is forbidden in service code (enforced by
ruff lint rule + code review).

### 2. Correlation ID model

The system uses W3C Trace Context (`traceparent` header / OTel propagation)
as the canonical correlation primitive. The `correlation_id` field is
distinct from `trace_id`:

- **`trace_id` / `span_id`**: per W3C; spans the lifetime of a single
  request or async operation. Used for distributed tracing.
- **`correlation_id`**: per **business record** (e.g., per ingested
  partner record). Threads through the entire pipeline regardless of
  service boundaries.

`correlation_id` is generated at the entry point:
- For partner file ingestion: per source-file (`source_file_correlation_id`)
  with per-row sub-correlation derived (`{source_file_id}:{row_index}`).
- For Verification API requests: per request, passed in a `Lore-Correlation-Id`
  header, generated by the load balancer if absent.

Propagation:
- **Pub/Sub messages**: `correlation_id` is a required message attribute.
- **Database rows**: an `originating_correlation_id` column on tables that
  represent results of pipeline processing (e.g., `match_decision`,
  `canonical_member_history`).
- **Audit events**: `correlation_id` is a required field in the audit event
  schema.
- **Log records**: bound to the logger context (via
  `structlog.contextvars.bind_contextvars`).

### 3. Error taxonomy

All services raise exceptions from a shared catalog in `shared/errors.py`.
The base hierarchy (already present from Phase 00) is extended:

```
LoreEligibilityError (base)
├── ConfigurationError       (already present — startup misconfiguration)
├── DataIntegrityError       (already present — invariant violation in stored data)
├── IdentityResolutionError  (already present — match cannot be made)
├── TransientError           (NEW — retry with backoff is the right response)
│   ├── DatabaseUnavailable
│   ├── KMSUnavailable
│   ├── PubSubPublishFailed
│   └── DownstreamServiceUnavailable
├── BackpressureSignal       (NEW — slow down upstream)
│   ├── RateLimitExceeded
│   ├── QueueDepthExceeded
│   └── KMSQuotaApproaching
├── PermanentDataError       (NEW — quarantine; do not retry)
│   ├── MalformedInput
│   ├── SchemaViolation
│   └── ContentRejection
├── PermanentSystemError     (NEW — page on-call; do not retry)
│   ├── InvariantBroken
│   ├── UnreachableState
│   └── DataCorruption
└── BusinessOutcome          (NEW — not an error in the system; e.g., NOT_VERIFIED)
    ├── NotVerified          (Verification API outcome)
    ├── PendingResolution    (Identity Resolution Tier 3)
    └── Quarantined          (DQ engine record-level rejection)
```

### 4. Retry policy by category

| Category | Retry? | Backoff | Max attempts | Terminal action |
|---|---|---|---|---|
| `TransientError` | Yes | Exponential, jitter | 5 | Quarantine + alert |
| `BackpressureSignal` | Yes (delayed) | Exponential, longer base | 3 | Pause upstream |
| `PermanentDataError` | No | — | — | Quarantine; do not alert (BR-302) |
| `PermanentSystemError` | No | — | — | Page on-call (P0) |
| `BusinessOutcome` | N/A | — | — | Return to caller; emit audit |

Implementation: a shared retry decorator in `shared/resilience.py` consumes
the exception type and applies the policy. No per-service inline retry logic.

### 5. Tracing

OpenTelemetry instrumentation in every service. Span boundaries:

- HTTP request (entry/exit at API boundary).
- Pub/Sub publish (span ends at publish; new span at consumer side, linked
  via trace context attribute).
- Database query (one span per query for queries > 10ms; sampled at 1% for
  faster queries).
- External service call (KMS, BigQuery, Splink, etc.).

Sampling: 100% for ingestion pipeline (low volume, high debugging value);
1% for Verification API (high volume); 10% for everything else.

### 6. Health probes

Per service:

- **`/livez`**: process is up. Always 200 unless terminating. Used by Cloud
  Run liveness check.
- **`/readyz`**: dependencies reachable (DB ping, Pub/Sub publish-permission
  check, downstream services reachable). 503 if any dependency unreachable.
  Used by Cloud Run readiness check (controls traffic routing).
- **`/health`**: existing endpoint; alias of `/readyz` for backward
  compatibility with existing harness.

### 7. Alert classification

| Severity | Definition | Routing |
|---|---|---|
| P0 | Service-down or security boundary breach | PagerDuty page on-call |
| P1 | Degraded but serving (SLO breach trend) | Page on-call within 1 hour |
| P2 | Operational degradation (DQ trend, backlog growth) | Notification, business-hours response |
| P3 | Informational | Logged, weekly review |

Every alert in PagerDuty / Cloud Monitoring is tagged with severity. Alert
quality (D-036) reviewed weekly: false positives retuned; outdated alerts
removed.

## Consequences

### Positive

- **Universal observability**: every log record has the same minimum schema
  across services.
- **Cross-service debugging is feasible**: `correlation_id` threads through
  the whole pipeline.
- **Consistent error handling**: shared taxonomy + retry decorator
  eliminates per-service drift.
- **PII redaction is universal**: harness-level redaction processors apply
  to every service.

### Negative

- **Discipline overhead**: developers must use the shared logger factory and
  shared error catalog. CI lint rules enforce; review enforces.
- **Context bookkeeping**: `correlation_id` propagation requires careful
  attention at every async boundary.

### Mitigations

- Shared library (`shared/telemetry.py`, `shared/errors.py`,
  `shared/resilience.py`) centralizes the conventions; developers consume
  the library, don't reinvent.
- Lint rules ban direct `logging.getLogger()` and direct `Exception` raise
  in service code.
- Tests enforce: every service's smoke test verifies that logs contain
  required fields.

## Alternatives considered

1. **Per-service logging conventions**: rejected. Inconsistency across services
   makes cross-service debugging impossible.
2. **Trace ID as the only correlation primitive**: rejected. W3C trace
   context is per-request; we need per-business-record correlation that
   spans many requests.
3. **Custom in-house tracing instead of OpenTelemetry**: rejected. OTel is
   the industry standard; vendor portability and ecosystem support are
   significant.
